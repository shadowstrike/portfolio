Program Overview

Confluent Manager for Apache Flink (CMF) handles Flink cluster deployment, job lifecycle management, RBAC integration, and licensing for Confluent Platform Flink. Initially, CMF only offered CLI and REST API interfaces - limiting accessibility for operators and developers who prefer graphical interfaces.

The CMF Web UI program introduced a standalone user interface integrated into Confluent Control Center, positioning CP Flink as a viable alternative to Confluent Cloud's fully managed Flink service while competing with open-source alternatives like Ververica Platform and Cloudera.

Strategic Context

Business Objectives

Address Usability Challenges

Flink Kubernetes Application Mode creates 1-1 mapping between jobs and clusters. Without UI, operators must surface information about entire fleets of running applications via CLI/API.

Reaccelerate Platform Adoption

Position CP Flink as premier self-managed alternative. Stimulate Confluent Platform uptake through useful management tooling.

Spearhead Stream Processing Innovation

Position Confluent as thought leader in evolving stream processing technologies and practices.

Competitive Analysis

CMF without UI executes Flink jobs in Application Mode via Flink K8s operator but limits management to REST API and CLI. This hides CMF behind poor UX compared to competitors:

  • Open Source Flink: Has web UI for job monitoring and deployment
  • Ververica Platform: Sophisticated UI for Flink deployment and monitoring
  • Cloudera: Web UI for Flink on YARN

Key Insight

Apache Flink has web UI for monitoring. CMF executes jobs in Application Mode which serves to hide functionality like uploading JAR and launching jobs. When run in Flink K8s application mode, the Flink UI cannot see all jobs in a K8s environment, necessitating higher-level UI.

UI Design Goals

Goal 1: Separate Development and Operations

Lower barrier for operations. Allow organizations to separate roles of developer and application administrators for Apache Flink. Make CMF and Apache Flink deployments more accessible to non-Flink specialists.

UI should offer ability to create, update, and destroy Flink environments and applications with minimal required specialized training.

Goal 2: Codify Operational Best Practices

Improve quality and reliability of Flink deployments, reducing level of error in Flink operations.

CMF addresses this by allowing administrators to specify application and environment-level defaults governing settings like resource consumption, parallelism, and Flink configurations. UI makes these settings available to less specialized operational staff without requiring alterations to application code.

Goal 3 (Long Term): Interactive SQL Development

Improve Confluent customer experiences and codify best practices with CP Flink by offering interactive SQL development workspace consistent with Confluent Cloud.

Confluent Cloud offers SQL interactive development environment and UDF support. Incorporation of standalone web UI would bring capability to CP Flink. A SQL web UI could lower skill level required to deploy Flink jobs.

Goal 4 (Long Term): Unify CC, C3, and CMF UX via Project Setu

Over time, incorporate CMF UI into broader Control Center and Confluent Cloud User Experience across Confluent.

User Personas Served

Primary Persona: Omar the Operator 📊

First revision of UI geared exclusively towards Flink cluster deployment and operations. The only persona served is Omar the Operator who manages the company's streaming infrastructure, ensures system health, and provides development teams with the tools they need.

User Flow 0: Licensing

Confluent wants Omar to provide and confirm valid Confluent Platform license so that Confluent can properly determine entitlement status.

  • Omar heads to licensing page
  • Page shows whether or not there is valid license
  • Omar clicks link to upload license or paste/upload valid license string

Core User Flows

  • Flow 1: Manage Flink Environments (view list, access details, create, update defaults)
  • Flow 2: Manage Flink Applications (view list, access details, launch, update configuration)
  • Flow 3: Fine-tune environment defaults and application configurations for real-world deployment, troubleshooting, and sizing

Early Design Exploration & Storyboarding

Initial exploration began with hand-drawn storyboards mapping out user journeys, interaction flows, and critical moments in the CMF Web UI experience. These sketches helped align stakeholders on key scenarios and identify edge cases early:

Hand-drawn storyboards exploring onboarding, environment creation, application management, and operator workflows

Why Storyboarding First?

Before jumping into high-fidelity mockups, I created these rough sketches to:

  • Map end-to-end user journeys across multiple screens and states
  • Identify dependencies between environments, applications, and licensing
  • Surface questions about error states, loading patterns, and progressive disclosure
  • Get early feedback from engineering on technical feasibility

This low-fidelity exploration saved weeks of iteration by validating core flows before investing in polished designs.

Design Philosophy

Throughout this program, maintained focus on progressive disclosure (simple defaults, advanced options available) and dual persona support (operators managing infrastructure, developers executing workloads).

Program Impact

Competitive Positioning

Closed UX gaps with Ververica Platform and Cloudera, removing major barrier in enterprise sales conversations.

Operational Accessibility

Enabled organizations to separate developer and operator roles - operators can manage Flink without Java/Scala expertise.

Platform Adoption

Lowered barrier to entry for Confluent Platform Flink adoption, particularly in regulated industries where CLI tools are restricted.

Strategic Outcomes

CMF Web UI in C3 positions CP Flink as viable self-managed alternative to Confluent Cloud. Organizations with security, compliance, or cost constraints can deploy Flink on-premises while maintaining cloud-like user experience.
