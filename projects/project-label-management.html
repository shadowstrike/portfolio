Label Management in CP Flink

Designed a comprehensive tag management, backend-powered filtering, and pagination system that eliminated a hardcoded 1,000-resource scalability cap and introduced Kubernetes-style key:value tagging across 7 resource types — unblocking UBS (200+ apps) and Medidata (1,000+ apps) from managing deployments at enterprise scale.

Overview

The C3 UI for CP Flink had two critical limitations that blocked enterprise adoption at scale. First, all REST API calls were hardcoded to a maximum of 1,000 resources, with all sorting and filtering performed entirely in the browser — creating silent data truncation for large deployments. Second, despite the CMF backend already supporting Kubernetes-style key:value labels, there was no label or tagging system in the UI at all.

This project addresses both: a comprehensive UI for tag management, backend-powered filtering and sorting, field projection, and pagination across all major CP Flink resource types — directly unblocking UBS (200+ apps, 50+ environments) and Medidata (1,000+ apps) who could not manage their deployments effectively through the C3 UI at their scale.

Application list view with tag chips visible inline — operators see resource groupings without navigating to detail pages

Context & Challenge

User Personas

Omar the Operator 📊

Platform Admin / Cluster Admin. Omar needs to organize and find resources at scale using tags, and filter large resource lists efficiently. He manages Flink deployments for multiple teams and needs to quickly identify which applications belong to which team, filter compute pools by environment tier, and audit resource usage across business units. Without tags, every resource lookup requires scrolling through truncated lists filtered only by name.

Dell the Lead Developer 👨‍💻

Flink Developer. Dell needs to quickly locate his applications, statements, and compute pools using name search or tag filters. He wants to tag his resources with team and project identifiers so he can filter list views to just his team's resources, and associate related resources through shared tag values.

The Problem

Customer Blockers

Two enterprise customers were specifically blocked by these limitations:

  • UBS — 200+ apps, 50+ environments. Need: multi-environment management at scale. The 1,000-resource cap and browser-side filtering made their deployments unmanageable through the UI.
  • Medidata — 1,000+ apps, few environments. Need: key:value label-based resource management. Without a tagging system, organizing 1,000+ applications was impossible.

Four compounding problems created this crisis:

  • Silent data truncation — The hardcoded 1,000-resource limit surfaced as incomplete resource lists with no indication to users that results were capped
  • Browser-side performance — All sorting and filtering by resource name ran entirely in the browser, making large deployments unusable
  • Feature parity gap — CLI and CFK already supported labels; the UI was the only surface that didn't, creating an inconsistent experience across Confluent's own tooling
  • Competitive gap — Ververica Platform and Decodable already supported k/v label management in their UIs

Design Scope

This design requirement (INIT-14197, RFC #54.2) covers label management, backend-powered filtering/sorting, field projection, and pagination across all major CP Flink resource types in C3.

In Scope

  • Tag CRUD (create, read, update, delete) across all supported resource types
  • Tag display in list views (table column) and detail pages
  • Tags in resource creation flows
  • Backend-powered tag filtering with autosuggestions
  • Backend-powered name search with wildcard support
  • Backend-powered sorting with sortable column headers
  • Pagination (replacing the hardcoded 1,000-resource limit)
  • Multi-field search for Environments, Secrets, and Events
  • Preserving all existing filters (status, phase, type, recency)
  • K8s label validation with inline error states

Out of Scope

  • Artifacts — covered separately in RFC #56 / INIT-6236
  • CLI changes for labels (deferred beyond Q1 2026)
  • REST API v1 → v2 migration (deferred beyond Q1 2026)
  • Bulk label operations (future enhancement)
  • Label colors (no backend support; considered for future)

Label Format — Kubernetes Semantics

Tags follow Kubernetes conventions: key (≤63 chars, alphanumeric start/end, optional DNS prefix up to 253 chars) + optional value (≤63 chars). Examples of real-world use cases:

  • team: datascience — organize by team ownership
  • environment: production — distinguish prod vs staging vs dev
  • cost-center: CC-4521 — map to billing/cost centers
  • platform.confluent.io/owner: analytics-team — prefixed key (K8s DNS subdomain convention)
  • temporary — key-only, acts like a simple boolean tag

Solution Design

Core Tag Interaction Pattern

Tag management is built around a consistent slide panel that appears identically across all resource types. Operators add tags using key-value pairs, with real-time Kubernetes validation, duplicate detection, autosuggestions from recently used keys/values, and toast notifications confirming every action.

Autosuggestion Strategy

The slide panel fetches suggested tag keys and values from the 200 most recently updated resources via the projection API (?fields=metadata.labels&size=200). Key:value pairs are collected into a multimap for typeahead/autocomplete in both Key and Value fields. With 500+ resources, not all possible label values will be shown — this is a best-effort approach.

Adding Tags

Add tag slide panel — initial state, key-value entry, K8s validation, duplicate detection, tooltips, and success notifications. Key field is read-only after creation (label keys are immutable once set).

Editing Tags

Edit tags — same slide panel, key field read-only (immutable), value field editable. Resources that don't support updates (Savepoints) show tags as read-only after creation.

Viewing All Tags

Overflow tag display — list views show a +N indicator when a resource has more tags than fit in the cell; clicking reveals the full tag set

Filtering, Sorting & Pagination

Beyond tags, this project moves all filtering and sorting from the browser to the backend — and replaces the hardcoded 1,000-resource cap with proper pagination. Every list view gains backend-powered capabilities.

Filter Combination Logic

  • Boolean AND — all tag filters, name search, and existing status/phase filters combine using AND
  • OR is supported only within multi-field search (Environments, Secrets, Events)
  • Set-based filters (in, notin) provide OR within a single label key

Tag Filter Syntax (User-Facing)

Users don't type raw filter syntax — the UI provides visual filter controls (dropdowns, chips) that translate to the ?filter= query parameter behind the scenes:

  • environment = production — equality: label key "environment" equals "production"
  • tier != frontend — inequality: label key "tier" is not "frontend"
  • environment in (production, qa) — set inclusion: environment is "production" OR "qa"
  • partition — existence: resource has a tag with key "partition"
  • !partition — non-existence: resource does NOT have key "partition"
  • region in (aws-*, azure-us-east2) — wildcard matching on values

Multi-Field Search

  • Environments: searches across name AND kubernetesNamespace (OR logic)
  • Secrets: searches across name AND status.environments (OR logic)
  • Events: searches across status.message AND metadata.flinkApplicationInstance (OR logic)
  • Multi-field search combines with tag filters via AND logic

Resource Coverage

Tags and the new filtering/sorting/pagination capabilities apply to all 7 major CP Flink resource types. Each resource has different constraints — notably, Compute Pool tag editing is pending backend support (CF-1688), and Savepoints have limited tag support until a details page exists.

Applications — List View

Application list view — tag chips inline, expandable overflow, backend sorting by name / namespace / specUpdatedTime / createdTime

Applications — Detail View

Application detail page — chips in header/metadata area with dismiss (×) button per chip; "+ Add tags" triggers slide panel

Compute Pools

Compute pools — tag filter dropdown and sortable columns. Note: tag edit/delete is pending backend support (CF-1688, targeting CMF 2.3 / April 2026) — design includes full edit flow gated behind backend availability check

Environments

Environments list — multi-field search (name + kubernetesNamespace), sortable by name and activeCount. Tags column with filter dropdown

Environment detail page — tags in header area with inline edit access; labels can be set during environment creation

Catalogs

Catalogs — tags work within environment-scoped views, maintaining context-awareness across the resource hierarchy

Secrets

Secrets — multi-field search across name + status.environments; tags identify ownership without exposing secret values

Statements

Statement list — sortable by name / creationTimestamp / updateTimestamp; tag filter + existing Phase filter preserved

Statement with tags applied — label chips visible inline on both list and detail views

Savepoints

Attached savepoints — tag display possible if a details page exists; sortable by name and creationTimestamp

Detached savepoints — tags supported on list and detail views; edit/delete not supported (no backend update support for savepoints)

Key Design Decisions

Several significant decisions were resolved during the design requirements phase. These are recorded here for future reference.

# Question Decision & Rationale
1 Should the UI use "Labels" or "Tags"? RESOLVED — Use "Tags". Ensures consistency with Confluent Cloud UI, which already uses "Tags" for resource metadata. The backend API uses metadata.labels but the UI-facing term prioritizes cross-product consistency for users who work across both CC and CP. The backend field name remains unchanged; this is a UI display-only decision.
2 Should label colors be supported in V1? RESOLVED — No. Multiple colors will not be supported. Confirmed by Robert Metzger (Backend Lead). No backend support exists; parked for a future release.
3 Should bulk label operations (apply tag to multiple resources) be supported? RESOLVED — No. Not in RFC scope; will not be supported in V1.
4 Should the slide panel be reusable across all resource types or customized per resource? RESOLVED — Yes, reusable. One slide panel design serves all resource types. Context is provided by the panel title: "Add Tag for <resource-name>" / "Edit Tag for <resource-name>".
5 Can we edit/delete tags for Compute Pools? Backend update support is pending (CF-1688). OPEN — Recommendation: Design the full edit/delete flow now and gate it behind a backend availability check. CMF 2.3 should land before C3 2.6 (June). Add a feature flag note for the C3 frontend team.
6 How should the filter bar scale when both tag filters and existing filters are shown? OPEN — Consider a unified filter bar with categorized dropdowns.
7 How do we communicate wildcard search syntax to users? OPEN — Proposal: Placeholder text + tooltip/info icon (e.g., placeholder "Search by name or namespace...").
8 How do we handle invalid label key/value typed in the filter? OPEN — Proposal: Show inline validation error and prevent filter submission.

Impact

Unblocked Enterprise Customers

Directly unblocked UBS (200+ apps, 50+ environments) and Medidata (1,000+ apps) — the two enterprise customers explicitly blocked by the 1,000-resource cap and lack of tagging. Enabled effective deployment management through the C3 UI at their scale.

Eliminated the 1,000-Resource Cap

Replaced hardcoded browser-side limits with backend-powered pagination and sorting, ending silent data truncation. Users now see complete resource lists with clear pagination controls — no more invisible caps on enterprise deployments.

7 Resource Types, One Interaction

Delivered tag management across all major CP Flink resource types using a single reusable slide panel, eliminating per-resource learning curves. Operators familiar with tagging one resource type immediately know how to tag any other.

Closed Three Gaps at Once

Achieved feature parity with CLI/CFK (which already supported labels), matched competitor UI capabilities (Ververica, Decodable), and unified terminology with Confluent Cloud — resolving a feature parity gap, a competitive gap, and an internal consistency gap simultaneously.

Key Learnings

Scalability Is a Design Problem, Not Just an Engineering Problem

The 1,000-resource cap was an engineering decision but its most visible impact was UX — users saw truncated lists with no indication that results were capped. Silent data truncation is a design failure. Surfacing honest system limits (or removing them via pagination) is as much a design responsibility as the interaction model.

Terminology Decisions Have Long-Term Costs

The "Labels" vs "Tags" debate revealed how terminology misalignment between backend APIs, CLI tooling, and UI creates lasting confusion for users who work across surfaces. The decision to use "Tags" in the UI (matching Confluent Cloud) while leaving the API field name as metadata.labels was pragmatic — prioritizing cross-product consistency over technical purity.

Design Insight

The most important design decision in this project wasn't about interaction patterns — it was deciding to scope all 7 resource types into a single release rather than incrementally. Tags only deliver organizational value when they're present everywhere. A partial rollout would have created a patchwork where some resources are taggable and others aren't, forcing users to maintain two organizational systems simultaneously. Consistency at launch was the right call.

Backend Dependency Risk Management

The Compute Pool tag editing situation (pending CF-1688) led to the approach of designing the full edit/delete flow and gating it behind a backend availability check rather than shipping a read-only-only design. Designing ahead of the backend ensures the full experience is ready when the API is, and avoids two separate design and engineering cycles.
