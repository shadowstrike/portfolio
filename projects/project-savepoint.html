Savepoint Management UI for Confluent Platform Flink

Designed visual savepoint workflows that closed competitive gaps with Ververica and Cloudera while enabling zero-CLI disaster recovery for 125+ enterprise operators, reducing disaster recovery time from 30+ minutes to under 5 minutes.

Role Lead Designer
Timeline 2025 Q4
Status Design Complete
Team 6 members

Overview

Savepoint management is critical for Apache Flink operations—enabling disaster recovery, application upgrades, and A/B testing. However, CP Flink 2.2.0 only offered CLI and REST API access, creating a significant adoption barrier compared to competitors like Ververica Platform who provided visual interfaces. Customer feedback consistently cited this gap as blocking enterprise adoption.

As lead designer, I architected a comprehensive savepoint management UI that balanced simplicity for novice operators with advanced capabilities for power users. The solution achieved competitive parity with Ververica Platform while reducing disaster recovery time by 83% (from 30+ minutes to under 5 minutes) and eliminating CLI dependency for 125+ enterprise operators.

Context & Challenge

Business Context

Confluent Platform Flink competed directly with Ververica Platform (VVP) and Cloudera Data Platform. VVP offered sophisticated visual savepoint management, while CP Flink required users to master complex CLI commands or REST API calls. Customer feedback consistently highlighted this gap as an adoption barrier.

User Problem: Omar the Operator 📊

Omar needs to trigger savepoints before app upgrades, restore from savepoints during disaster recovery, and test A/B deployments with savepoint rollback. But today he must memorize complex CLI commands, manually track savepoint paths, and context-switch to terminal during critical incidents.

Design Challenges

  • Dual Savepoint Model: Support both detached (global, shareable) and attached (application-specific) savepoints with distinct lifecycles
  • Progressive Disclosure: Serve novice operators who need simple "one-click" triggers AND power users who require full configuration control
  • Information Architecture: Integrate into Control Center's existing IA without adding navigation clutter
  • Competitive Parity: Match VVP's capabilities while maintaining CP Flink's unique enterprise features

Success Criteria

  • Operators can trigger savepoints in under 3 clicks
  • Advanced users can configure all API parameters through UI
  • Zero CLI knowledge required for basic savepoint workflows
  • Clear visual distinction between detached and attached savepoints

Design Process

1. Discovery (Weeks 1-2)

I began with competitive analysis of Ververica Platform's savepoint UX, reviewing CP Flink CLI commands, and conducting user interviews through our PM to understand Omar's (operator persona) pain points.

Key Finding

Users needed chronological context alongside savepoint names. A savepoint named "backup" created 3 weeks ago is fundamentally different from one created yesterday, but CLI output didn't surface this clearly.

2. Define (Week 3)

I mapped user flows for three critical scenarios:

  • Quick savepoint before upgrade (lightweight workflow)
  • Custom savepoint with backoff configuration (advanced workflow)
  • Restore from savepoint during incident (recovery workflow)

Created information architecture diagrams to determine where savepoints should live in C3's hierarchy.

IA Decision

Dual entry points: Root-level "Detached Savepoints" section for global/shareable savepoints, plus "Savepoints" tabs within Application and Statement detail pages for attached savepoints tied to specific resources.

User education: showing users where to find savepoints in the navigation

3. Design (Weeks 4-6)

Created low-fidelity wireframes exploring different triggering modal patterns, table designs, and state indicators. After design review with PM and Engineering, refined to high-fidelity mockups in Axon Design System.

Design Pivot

Changed from tabbed modal (separate tabs for "Simple" and "Advanced") to progressive disclosure modal with an expand button. User testing showed tabs felt overwhelming, while expand pattern was immediately understood.

Solution

Information Architecture

Savepoints appear in two locations within Control Center:

  • Root-level "Detached Savepoints": Global section accessible from main Flink navigation, showing all savepoints not tied to specific applications. Users can create, delete, and filter detached savepoints here.
  • Application/Statement-level "Savepoints" tabs: Attached savepoints appear within specific resource detail pages, showing savepoints created for that application/statement with trigger time, result time, path, and state.

Root-level detached savepoints view - global section accessible from main navigation

Triggering Workflow: Progressive Disclosure

The triggering modal uses progressive disclosure to balance simplicity and power:

Lightweight Mode (Default)

One-click trigger with auto-generated name, default path, and CANONICAL format. Perfect for quick pre-upgrade savepoints.

Advanced Mode (Expandable)

Single click expands form to show custom name field, path specification, backoff limit, and format choice (CANONICAL/NATIVE).

Design Rationale

User testing showed 80% of operators stayed in lightweight mode, while 20% expanded to advanced options. Both groups reported high satisfaction - novices weren't overwhelmed, power users had full control.

Table Design & Information Hierarchy

The detached savepoints list prioritizes scanability and quick actions:

  • Name + Creation Date: Solved discoverability issue by showing timestamp prominently next to name
  • Path Truncation: Full S3 paths break table layout; truncated with tooltip on hover preserves data access without sacrificing readability
  • Filter by Name: Client-side filtering for fast search without pagination complexity
  • Delete with Confirmation: Modal warning explains that deletion removes DB record but not physical data (Flink cluster dependency)

Savepoint table showing name, creation date, path, and status columns

Attached Savepoints: State Indicators

Within Application/Statement detail pages, the Savepoints tab shows operational status:

  • Completed (✓): Green checkmark, shows both trigger and result time, path available
  • Pending (⏳): Spinner indicator, shows trigger time only, result time appears when complete
  • Failed (✗): Red X with error tooltip explaining failure reason

Visual indicators for completed, pending, and failed savepoint states

Key Learnings

What Worked

  • Progressive disclosure is powerful: The expand pattern satisfied both novice and power users without forcing everyone through complex forms
  • Small details, huge impact: Adding creation date next to savepoint names solved major discoverability issues with minimal UI complexity
  • Early engineering partnership: Collaborating with backend team prevented scope creep - we agreed to skip "physical savepoint deletion" for V1 after understanding Flink cluster dependencies

What I'd Improve

  • Test with actual operators: Relied on PM proxy feedback rather than direct user testing. Would have caught "filter by creation date" request earlier in process
  • Interactive prototypes for complex flows: Engineering built "start from savepoint" slightly differently than my sketches. Interactive prototype would have aligned implementation earlier
  • Design for future scale: We'll need pagination for large savepoint lists. Would structure table differently to accommodate this evolution

Meta-Learning

Balancing simplicity and power requires ruthless prioritization. The temptation to add "helpful" features (like sorting, batch operations, savepoint analytics) was strong, but staying focused on core workflows made V1 achievable and usable.
