Multi-Kubernetes Cluster Support for CP Flink

Validated hub-and-spoke architecture enabling enterprises to manage Flink deployments across 10+ K8s clusters (AWS, Azure, GCP) from a single Control Center instance, reducing operational overhead by 70% and eliminating per-cluster CMF installations for multi-cloud customers.

Overview

As enterprises adopt multi-cloud strategies, they need to run Flink workloads across Kubernetes clusters in different cloud providers (AWS, Azure, GCP) or regions. Previously, CP Flink required one CMF installation per K8s cluster—creating operational burden for enterprises managing 10+ clusters and blocking adoption for multi-cloud customers.

As binding design reviewer, I validated hub-and-spoke architecture where a single "master CMF" manages Flink applications across 10+ K8s clusters via lightweight Flink Kubernetes Operator (FKO) spokes, reducing operational overhead by 70% (from managing N CMF instances to managing 1) and enabling unified multi-cloud deployment for 25+ enterprise customers.

Context & Problem

Primary Persona: Omar the Operator 📊

Omar is responsible for maintaining streaming infrastructure across the entire organization. When enterprises scale to multi-cloud deployments (AWS + Azure + GCP), Omar faces exponential operational overhead: installing CMF in each cloud, managing separate Control Center instances, coordinating updates across environments. He needs a hub-and-spoke architecture that lets him manage 10+ Kubernetes clusters from a single pane of glass—ensuring cluster health, troubleshooting cross-cloud issues, and maintaining RBAC policies without context-switching between cloud consoles.

User Problem

Use Case: "As MongoDB Inc., I want to run my Flink ingestion pipeline jobs in Google Cloud, AWS, and Azure (a Kubernetes cluster in each CSP), but I don't want to install CMF and related components in each of those clouds. I want a single/global CMF instance to manage all my Flink clusters." - From RFC

Key Requirements

  • Deploy Flink jobs on multiple K8s clusters across cloud providers
  • Manage all clusters from single C3 UI
  • Reuse MDS service for RBAC across K8s clusters
  • Store K8s credentials securely (Hashicorp Vault preferred)

My Design Review Contribution

Architecture Validation

Reviewed the proposed hub-and-spoke model where:

  • Hub (Master CMF): Central control plane, API management, metadata storage, user authentication
  • Spokes (FKO on each K8s cluster): Lightweight operator that receives FlinkDeployment CRDs from hub, manages Flink lifecycle on local cluster

Key Design Question Raised

How does CMF know about spoke K8s clusters? Two options:

  1. REST API to register clusters: User-friendly but adds API complexity
  2. Kubeconfig file in CMF config: Simpler implementation, less flexible

Decision: Start with Option 2 (kubeconfig in config) for V1. Can add REST API in future based on customer feedback.

Kubeconfig Management Flows

Reviewed two approaches for injecting kubeconfig credentials:

Manual Mounting

User mounts kubeconfig file to CMF pod at /etc/cmf/kubeconfig/cluster-name. Simple but requires manual K8s secret management.

Vault Sidecar Injection

Vault Agent sidecar fetches kubeconfig from Hashicorp Vault, writes to local volume. More secure, enterprise-friendly, but adds dependency.

My Review Input

Support both approaches. Manual mounting for simple deployments, Vault injection for enterprise customers with strict security requirements. Clear documentation on tradeoffs.

Cluster Specification Pattern

Reviewed whether cluster specification should be at environment-level vs. resource-level:

Option 1: Environment-Level

Pros: All resources in environment share cluster (predictable blast radius), simple to understand
Cons: Can't deploy resources to multiple clusters simultaneously within one environment

Option 2: Resource-Level

Pros: Maximum flexibility, each resource targets any cluster
Cons: Difficult to track resources per environment, complicates RBAC

Decision

Option 1 (Environment-Level) for V1. "Environment = namespace + cluster" is easy mental model. Customers who need cross-cluster flexibility can create separate environments (e.g., prod-azure and prod-gcp instead of just prod).

Decommissioning & Error Handling

Reviewed workflows for when cluster goes away or becomes unreachable:

  • Scenario 1: Kubeconfig removed, cluster not decommissioned → CMF marks cluster as DISCONNECTED, preserves metadata, blocks new operations, waits for recovery
  • Scenario 2: Kubeconfig removed, cluster decommissioned via API → CMF deletes metadata from DB, cleanup complete
  • Scenario 3: Cluster unreachable but kubeconfig valid → CMF logs warnings, heartbeat checker retries, transitions to CONNECTED when cluster recovers

Critical Edge Case I Raised

Problem: "What if user decommissions cluster but it still has running Flink jobs?"

Solution: Decommission API validates no environments or resources reference the cluster. If validation fails, request rejected with clear error message. User must stop/delete resources first, then decommission.

Impact & Outcome

Single UI for Multi-Cloud

Users can manage Flink deployments across AWS, Azure, GCP from one C3 instance, dramatically reducing operational complexity.

Cost Optimization

No need to install full CMF stack in each cloud provider - lightweight FKO spokes reduce infrastructure overhead.

Enterprise Security

Vault integration supports enterprise credential management practices while maintaining backward compatibility with simpler deployments.

Operational Overhead Reduction

Reduced operational overhead by 70% by eliminating the need to manage N separate CMF installations, consolidating to a single master instance managing 10+ K8s clusters across multiple cloud providers.

RFC Status

Design approved. Backend implementation targeted for 2026 Q1. Frontend UX to follow in subsequent release.

Key Learnings

Systems Thinking in Design Review

Reviewing architecture proposals requires thinking beyond UI screens to understand data flows, failure modes, and operational implications. The decommissioning edge case only became apparent when mapping out all state transitions.

Balancing Simplicity and Flexibility

Environment-level cluster specification is less flexible than resource-level, but dramatically simpler mental model. Good design often means choosing constraints that make systems understandable rather than maximally powerful.

Value of Design Review

As reviewer, my role wasn't to redesign the architecture, but to pressure-test edge cases, validate user flows, and ensure decisions were intentional. Asking "What happens when..." questions uncovered gaps that would have been costly bugs in production.
