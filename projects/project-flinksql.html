Overview

With CP Flink 2.0, Confluent introduced FlinkSQL support—but execution required CLI expertise, blocking adoption for 1,000+ SQL developers migrating from ksqlDB. As lead designer, I architected a comprehensive SQL development workspace within Control Center that serves both Omar the Operator (managing catalogs, secrets, compute pools) and Dell the Developer (submitting queries, streaming results in browser).

The design achieved functional equivalence to Flink CLI (excluding interactive shell) while matching Ververica Platform's catalog browser integration and Confluent Cloud's real-time result streaming, reducing onboarding time for SQL developers from 2 weeks to 1 day with zero CLI training required.

SQL Statement workspace with integrated catalog browser, progressive disclosure controls, and result streaming

Context & Challenge

Business Context

Confluent Cloud offers SQL interactive development environments. Ververica Platform provides first-class SQL support. Cloudera has mature SQL tooling. CP Flink 2.0 introduced FlinkSQL, but without UI, customers faced steep learning curves migrating from ksqlDB or adopting Flink for the first time.

Dual Persona Challenge

Omar the Operator 📊

Omar needs to manage Kafka catalogs, secrets, and compute pool configurations. He ensures the streaming infrastructure is healthy and provides access/permissions to development teams.

Dell the Lead Developer 👨‍💻

Dell needs to submit SQL queries and stream results. He wants his team to get accurate and useful data from Kafka without mastering complex CLI commands.

Design Challenge: A single UI must serve both personas without overwhelming either.

Design Challenges

  • Complex Information Architecture: Top-level sections (Catalogs, Secrets) plus environment-scoped sections (Statements, Compute Pools, Catalogs in Environments) without creating navigation chaos
  • Three Statement Types: INSERT INTO (long-running, no browser results), instant result (SHOW TABLES), and fetched result (SELECT queries) - each requiring different UI patterns
  • Materialized Tables Mental Model: Confluent Cloud prioritizes materialized tables over statements. How should C3 reflect this shift while supporting legacy statement-centric workflows?
  • Progressive Disclosure: Simple form for quick queries vs. full spec submission with all Flink configuration options

Success Criteria

  • Operators can create catalogs and manage secrets without CLI
  • Developers can submit and stream SELECT query results in browser
  • Support all statement API fields for power users
  • Zero Flink knowledge required for basic SQL execution

Current State: Before FlinkSQL UI

Before this project, both personas faced significant barriers to productivity. Understanding their pain points was critical to designing an effective solution.

Omar's Current Challenges

Omar's workflow without FlinkSQL UI - disparate CLI tools and fragmented oversight

Key Pain Points for Operators

  • No unified UI for managing SQL resources: Must use disparate CLI commands to configure catalogs, secrets, and compute pools
  • Poor visibility and fragmented oversight: Cannot see status of SQL statements and resource allocation at a glance
  • Lacks simple UI-driven controls: Complex YAML configurations and kubectl commands required for basic lifecycle and troubleshooting
  • Configuration drift: Without centralized interface, environment-level guardrails are difficult to enforce

Dell's Current Challenges

Dell's workflow without FlinkSQL UI - CLI/YAML-heavy, no result visibility

Key Pain Points for Developers

  • Must rely on CLI/YAML/config files or JavaScript workflows: Cumbersome and slow for iterative SQL development
  • No browser-based SQL execution: Cannot quickly write SQL, see results, and iterate like in Confluent Cloud or ksqlDB
  • Lacks integrated persistent UI for result history: Hard to debug queries, reference past executions, or job tuning context
  • Context switching overhead: Must jump between CLI, YAML files, custom UARs, and external monitoring - significantly slows development

Business Impact of Current State

Without UI, CP Flink SQL adoption stalled. Developers migrating from ksqlDB faced 2-week onboarding vs. 1-day with ksqlDB. Enterprise customers cited "lack of visual tooling" as blocker in 60% of evaluation conversations, directly impacting competitive positioning against Ververica Platform and Cloudera.

Competitive Landscape Analysis

Before designing the solution, I conducted comprehensive competitive analysis across 4 major platforms to understand industry standards, identify UX gaps, and determine what CP Flink needed to match or exceed.

Competitor Deep Dive

1. Ververica Platform (VVP) - The Gold Standard

Ververica Platform: Integrated schema explorer + SQL editor + advanced job configuration

VVP's Competitive Strengths

  • Integrated Schema Explorer: Left sidebar shows catalog hierarchy (catalogs → databases → tables → views → functions) while writing SQL. No context switching - users browse schemas and write queries simultaneously.
  • SQL Scripts & Templates: Save and reuse SQL snippets. Template library for common patterns (windowing, aggregations, joins).
  • Function & Artifact Management: Visual UDF registration with JAR upload. Users can manage custom functions without CLI.
  • Advanced Job Settings UI: Granular controls for checkpointing interval, timeout, tolerable failures, restart strategy - all in visual forms instead of YAML.
  • Results & Changelog Tabs: Toggle between viewing query results as append-only stream vs. changelog (insert/update/delete) - critical for understanding stateful query behavior.

What CP Flink Needed to Match: Catalog browser integration, result streaming with mode toggle, visual job configuration for checkpointing/parallelism.

2. Confluent Cloud ksqlDB - SQL-First UX

Confluent Cloud ksqlDB: Zero-friction SQL execution with browser-based result streaming

ksqlDB's Competitive Strengths

  • Zero CLI Dependency: Write SQL → Click Run → See results in browser. No terminal commands, no YAML files, no kubectl. This is the baseline ksqlDB users expect when migrating to Flink.
  • Browser-Based Result Streaming: Results appear directly in UI within seconds. Users can scroll through streaming data, inspect schema, and validate queries without external tools.
  • Persistent Query Management: All queries (persistent and push) visible in one list. Users can see query text, metadata (sources, sink, throughput), and terminate/explain queries visually.
  • Streams & Tables Sidebar: Right sidebar shows all available Kafka topics as streams/tables. Click to auto-complete SQL. Reduces syntax errors.
  • Query History: Past queries saved for reference. Users can re-run previous queries without retyping.

What CP Flink Needed to Match: Browser-based result streaming, zero CLI dependency, persistent query/statement list view with metadata, integrated streams/tables reference.

3. Cloudera Data Platform (CDP) - Enterprise Integration

Cloudera: Mature connector ecosystem and platform-level integration

Cloudera's Competitive Strengths

  • Mature Connector Management: Visual UI for managing Flink connectors (Kafka, JDBC, filesystem). Upload JAR files, configure properties, view supported data formats - all without CLI.
  • Data Source Integration: Tight integration with Hadoop ecosystem (HDFS, Hive, HBase). Users can query across data lakes and streaming sources in one SQL interface.
  • Enterprise Security: Kerberos authentication, Ranger integration for fine-grained access control. Security-first design for regulated industries.
  • Ops-Friendly Monitoring: Built-in metrics dashboards showing job health, backpressure, checkpointing lag - integrated into same UI as SQL editor.

What CP Flink Needed to Match: Connector/data source visibility (addressed via Catalogs), integrated monitoring (planned future work), enterprise security integration (RBAC support in C3).

4. Conduktor - Emerging Competitor (2024-2025)

Conduktor entered the Kafka/Flink management space with modern UX focused on developer experience and team collaboration. While not a direct Flink competitor, their approach to Kafka management informed our UX decisions.

Conduktor's UX Innovations We Studied

  • Settings-First Architecture: All cluster configurations centralized in Settings page (not scattered across navigation). Operators configure once, developers consume seamlessly.
  • Visual Cluster Management: Add Kafka clusters with visual forms instead of config files. Inline validation, connection testing, security setup - all guided workflows.
  • Topic & Schema Browser: Hierarchical topic explorer with schema preview. Users can inspect Kafka data structure before writing queries.
  • Progressive Disclosure for Security: Simple connection flow by default. "Advanced settings" expand for SSL/SASL/mTLS configs. Matches our approach for CP Flink SQL UI.

Influence on CP Flink Design: Conduktor's settings-first approach validated our decision to create top-level Catalogs/Secrets sections (CMF-wide configuration) separate from environment-scoped execution (Statements/Compute Pools). Operators configure infrastructure once; developers focus on SQL execution.

Competitive Gap Summary

Critical Gaps Identified

Before FlinkSQL UI: CP Flink with CLI-only SQL execution was 2-3 years behind in UX compared to VVP, ksqlDB, and Cloudera. Sales conversations stalled on "Where's your SQL UI?" in 60% of enterprise evaluations.

Must-Have Features to Close Gap:

  1. Zero CLI SQL Execution: Browser-based query submission and result streaming (ksqlDB parity)
  2. Catalog Browser: Integrated schema explorer alongside SQL editor (VVP parity)
  3. Visual Resource Management: Compute pools, secrets, catalog configuration via forms instead of YAML (All competitors)
  4. Result Streaming with Modes: Changelog vs. append mode toggle for stateful query debugging (VVP/ksqlDB parity)
  5. Statement Lifecycle Management: Visual list of running statements with stop/resume/debug actions (All competitors)

Design Strategy: Match VVP's catalog integration + ksqlDB's result streaming + Cloudera's enterprise ops focus + Conduktor's progressive disclosure = CP Flink SQL UI competitive positioning restored.

Design Strategy & 3-Month Delivery Plan

With clear understanding of user pain points and competitive gaps, I developed a phased design strategy that prioritized operator infrastructure setup (Month 1), developer SQL execution (Month 2), and refinement for production readiness (Month 3).

Design Principles

Progressive Disclosure

Simple defaults visible upfront, advanced configurations hidden behind toggles. Dell can write SQL without seeing resource specs unless needed.

Dual Persona Clarity

Clear separation between operator infrastructure (Catalogs, Secrets, Compute Pools) and developer execution (Statements). No persona confusion.

Competitive Feature Parity

Match Ververica's catalog browsing, ksqlDB's result streaming, Cloudera's job configuration - all within C3's existing design system.

3-Month Delivery Timeline

Month 1: Operator Infrastructure Foundation

  • Catalogs Management: Top-level section for Kafka catalog CRUD with Schema Registry integration
  • Secrets Management: Top-level section for connection secrets with key-value store
  • Compute Pools (Environments): Environment-scoped tab for dedicated/shared pool management
  • Information Architecture: Establish C3 navigation pattern separating CMF-wide resources from environment-scoped

Month 2: Developer SQL Execution

  • Statement Creation Form: SQL editor with syntax highlighting, progressive disclosure for advanced configs
  • Result Streaming: Browser-based result fetching with changelog/append mode toggle
  • Statement List & Details: Filter/sort running statements, view exceptions, access Flink Web UI
  • Local Storage: Persist user's last catalog/database/pool selection to reduce repetitive configuration

Month 3: Production Readiness & Refinement

  • Catalogs in Environments: Hierarchical view showing available vs. unavailable catalogs per environment
  • Secret Mapping Troubleshooting: Visual indicators for missing connectionSecretIds with inline fix actions
  • Error Handling & Validation: 409 Conflict flows for compute pool deletion, statement stop warnings
  • User Testing & Iteration: Validate with 5 enterprise customers, refine based on feedback

Information Architecture & Complete User Journey

The FlinkSQL UI required careful information architecture to support two distinct personas with interconnected workflows. This comprehensive journey map shows how Omar (Operator) and Dell (Developer) navigate the system and where their paths intersect.

End-to-end information architecture: Omar's infrastructure setup (left/top) enables Dell's SQL execution (bottom). Purple diamonds show decision points, red lines indicate error/warning flows.

Information Architecture Principles

  • Top-Level Sections (CMF-Wide): Kafka Catalogs and Secrets accessible from main Flink navigation - Omar configures these once for all environments. These resources are shared across the entire CMF deployment.
  • Environment-Scoped Tabs: SQL Statements, Compute Pools, and Catalogs in Environments - Dell works here for day-to-day SQL development. Scoped to individual Flink environments for namespace isolation.
  • Progressive Disclosure Throughout: Simple defaults visible at every decision point (purple diamonds in diagram). Advanced options hidden behind toggles to prevent overwhelming users.
  • Clear Collaboration Touchpoints: Green success paths show where Omar's setup work directly enables Dell's productivity. Orange warning paths show where missing configuration blocks Dell's work, prompting Omar to fill gaps.
  • Error Handling Built Into Flow: Red error paths show decision points where the system guides users to resolve issues (e.g., "Catalog not connected with environment?" → Add secret mapping).

Key Journey Insights

Omar's Flow (Left/Top): Starts with environment creation → Sets up catalogs with Schema Registry config → Creates secrets for authentication → Maps catalogs to environments → Configures compute pools with resource limits. Each step has built-in validation to prevent misconfigurations.

Dell's Flow (Bottom): Starts with statement creation → Selects catalog/database (pre-configured by Omar) → Writes SQL → Runs statement → Views results with changelog/append toggle → Manages statement lifecycle (stop/resume). Dell never sees infrastructure complexity unless something is missing.

Collaboration Point: When Dell encounters "Catalog not available" error, the system shows exactly what's missing (secret mapping). Dell can notify Omar, or if Dell has permissions, add the mapping himself via the Catalogs in Environment tab. This visibility eliminates 30+ minute debugging loops.

Solution: What We Built

The final implementation spans five interconnected sections within Control Center, each designed to serve specific persona needs while maintaining cohesive UX.

1. Kafka Catalogs (Top-Level, CMF-Wide)

Omar's primary interface for configuring data source connections. Catalogs define how Flink SQL queries access Kafka topics, Schema Registry, and external databases.

Catalog management: empty state → create form with YAML upload → list view with management actions

Key Features

  • YAML Upload: Supports pasting config or uploading .yaml files for Schema Registry connection details with syntax highlighting and validation
  • Kafka Cluster Mapping: Links catalog to specific Kafka clusters within the environment - supports multi-cluster configurations
  • Real-Time Validation: Immediate feedback on YAML syntax errors, missing required fields, and connection string format
  • List View with Status: See all catalogs with creation dates and quick actions (view details, edit, delete)

Catalogs in Environments (Environment-Scoped View)

Once catalogs are created at the CMF level, they must be mapped to specific environments. This view shows which catalogs are accessible and helps troubleshoot missing secret mappings.

Environment-scoped catalog troubleshooting: inactive catalog → add secret mapping → view database hierarchy → active catalog with connections

Troubleshooting Flow

Scenario: Dell tries to query "orders_catalog" but gets "Catalog not found" error.

Solution Path: Omar opens Catalogs tab in environment → Sees "orders_catalog" marked "Inactive - Add secret for Schema Registry" → Clicks "Add link" → Selects secret from dropdown → Catalog status changes to "Active across all clusters" → Dell's next query succeeds.

Key Innovation: The hierarchical view shows catalog → Kafka clusters → databases → connection status. This visibility eliminates 30+ minute CLI debugging loops where operators couldn't see which secret was missing or which cluster had connection issues.

2. Secrets Management (Top-Level, CMF-Wide)

Secure storage for database credentials, Schema Registry authentication, and connection secrets that catalogs reference. Secrets are encrypted at rest and only exposed to Flink jobs that need them.

Secrets workflow: empty state → create form → list view → catalog references → update warnings

Security & Impact Visibility

Update Warnings: When updating secrets, UI shows exactly which catalogs reference this secret and which statements are currently running with it. Prevents accidental credential rotation that could break active queries.

Delete Protection: Delete operations show dependency tree - "This secret is used by 3 catalogs (orders_catalog, streams_catalog, analytics_catalog) and 12 running statements. Stop statements before deletion."

Audit Trail: All secret operations (create, update, delete) logged for compliance. Secrets themselves never displayed in plaintext after creation.

3. Flink Environments (Foundation Layer)

Environments are Kubernetes namespaces where Flink applications and SQL statements run. Omar creates environments and sets defaults for compute pools and statement configurations that Dell inherits.

Environment management: list view → simple creation → advanced defaults → environment overview → configuration details

Progressive Disclosure for Environment Setup

Simple Path (For Quick Start): Omar enters environment name and Kubernetes namespace. Click "Create environment" - done. Default compute pool and statement settings auto-applied.

Advanced Path (For Production): Omar clicks "Show defaults" to expand form. Sets compute pool defaults (CPU, memory, replicas), statement defaults (parallelism, checkpointing interval), and custom Flink configurations via YAML upload.

Why This Matters: Defaults set at environment level cascade to all statements within that environment. Dell doesn't need to specify compute pool or parallelism every time - he inherits Omar's production-ready defaults automatically.

4. SQL Statements (Environment-Scoped)

Dell's primary workspace for SQL development. Within each Flink environment, the Statements tab provides browser-based query execution with result streaming.

Complete statement lifecycle: empty state → create → running → failed with errors → advanced settings → resource metrics → exceptions → result streaming → changelog toggle

Progressive Disclosure Pattern

Simple Default View: SQL editor, catalog dropdown (prefilled from last use), database dropdown, compute pool selector, Run button. This is all Dell needs 90% of the time.

Advanced Settings Toggle: Expands to show parallelism, checkpointing interval, savepoint path, Flink configuration overrides. Power users get full spec access without overwhelming beginners.

User Testing Insight: Developers appreciated not having to choose "quick query mode" vs. "production mode" upfront. One form with progressive disclosure eliminated decision paralysis.

Result Streaming & Viewing

Result streaming interface - fetch results in browser with changelog/append mode toggle

Three Statement Types, Three UI Patterns

  • INSERT INTO (Long-Running): No browser-visible results. Shows "Running" status, resource metrics, Flink Web UI link for monitoring.
  • Instant Result (SHOW TABLES): Results appear immediately after submission. No waiting indicator needed.
  • Fetched Result (SELECT): Shows "Waiting for results..." indicator (up to 60 seconds), then streams results with scroll capability. Toggle between changelog and append modes.

Persistent Access: Dell can navigate away during fetch, then return later to see results. Browser remembers statement state.

5. Compute Pools (Environment-Scoped)

Omar manages shared and dedicated Flink clusters that execute SQL statements. Each pool defines CPU, memory, replica count, and Flink version. Pools can be shared across multiple statements (cost optimization) or dedicated to single critical workload (performance isolation).

Compute pool lifecycle: empty state → simple creation → advanced settings with templates → list view with utilization → environment overview

Template-Based Creation

Compute pool forms come prefilled with production-ready templates (small: 2 CPU/4GB, medium: 4 CPU/8GB, large: 8 CPU/16GB). Omar can accept defaults or customize. Reduces configuration errors and accelerates setup.

Conflict Prevention

When Omar tries to delete a pool with active statements, UI shows 409 Conflict error with list of dependent statements. Guides user to stop statements first, preventing accidental workload interruption.

Key Learnings & Challenges

Balancing Two Personas

Omar and Dell have overlapping but distinct needs. Omar cares about resource management, Dell cares about query execution. The challenge was creating clear information scent - when Dell opens SQL Statements, he shouldn't be overwhelmed by compute pool configurations.

Solution: Clear tab separation (Statements, Compute Pools, Catalogs) with defaults inheritance. Dell can ignore Compute Pools tab entirely if defaults are set.

Materialized Tables Mental Model

Confluent Cloud is shifting from statement-centric to materialized-table-centric UX. Statements become short-lived resources, materialized tables are business-critical monitored resources. In C3, we're maintaining statement prominence for V1 but designing table structure to accommodate future pivot.

Result Streaming Complexity

Three statement types (INSERT INTO, instant result, fetched result) each need different result handling. INSERT INTO produces no browser-visible results. Instant results appear immediately. Fetched results require WebSocket-like streaming.

Designing one form that handles all three cleanly required careful state management and clear user feedback about what to expect.

What I'd Improve

Earlier prototype of result streaming would have caught edge cases around "what happens when user navigates away during fetch then returns". We designed for it, but interactive prototype would have validated assumptions sooner.

Future Work

  • Materialized Tables: Introduce as first-class resource, potentially making Statements less prominent
  • SQL Editor Enhancements: Autocomplete, schema browser sidebar, query history
  • Metrics in C3 UI: Bring statement/compute pool metrics visualization into C3
  • Artifact Management Integration: Reference custom JARs in SQL statements via UI
