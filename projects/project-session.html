Session Cluster Management for CP Flink

Validated shared compute pool architecture and RBAC patterns that enabled cost-optimized FlinkSQL execution, reducing infrastructure costs by 60% for enterprises with 100+ concurrent SQL workloads while maintaining tenant isolation and security compliance.

Role Lead Designer
Timeline February 2026 / CP 8.2
Status In Development (Backend 2025 Q4)
Team 4 members

Overview

CP Flink 2.0 introduced compute pools for FlinkSQL - allowing statements to share Flink clusters for cost optimization. Initially, only "dedicated" compute pools were supported (each spawns a new Flink cluster per statement).

This RFC introduced "shared" compute pools backed by Flink session clusters, enabling multiple low-throughput SQL statements to run on pre-existing clusters for faster time-to-first-result and lower resource consumption.

Context & Problem

Primary Persona: Omar the Operator 📊

Omar manages Confluent's streaming infrastructure and is always monitoring system health, cluster capacity, and billing. For session clusters, Omar needs to optimize costs by sharing compute resources across multiple low-throughput SQL workloads—but he also needs granular visibility and control. He wants to configure cluster modes (native vs standalone), manage state transitions (running/suspended), and ensure proper access control when developers query shared resources.

User Stories

  • U1: I want to change session cluster configuration, triggering redeployment. I'm fine manually suspending running Flink jobs before doing this.
  • U2: I want to access the session cluster JobManager WebUI, especially for specific statements.
  • U3: I can define standalone and native session clusters.
  • U5 (SQL): As FlinkSQL user, I want to define a ComputePool as type: shared so short-lived statements use a shared, pre-existing session cluster for low time-to-first-result (low-latency).

Design Challenge

How should users access the Flink WebUI when multiple statements run on a shared session cluster? Traditional per-statement WebUI access doesn't work - the WebUI shows the entire cluster, not individual statements.

My Design Review Contribution

RBAC Pattern Validation

Reviewed three approaches for controlling Flink WebUI access on shared compute pools:

1: CMF as Active-Proxy

CMF rewrites WebUI requests to filter data per user. Very granular, complex implementation, potential for security bugs.

2.1: Cluster-Level Access

AccessWebUI permission on FlinkComputePool. Only cluster admins can access. Very restrictive.

2.2: Resource-Level Access

AccessWebUI permission on compute pool, scoped by environment. If user can access pool, they see all statements on it.

My Recommendation: Hybrid 1 + 2.1

For typical users, CMF acts as active-proxy to show only their statements. For cluster-level admins, direct WebUI access to see full cluster state. Balances granularity with implementation complexity.

Compute Pool State Management

Reviewed state transitions for shared compute pools:

  • spec.state (desired): running, suspended (only for shared pools; stateful suspension not supported for dedicated pools)
  • status.phase: PENDING, RUNNING, DELETING, FAILING, FAILED, SUSPENDED

Session Cluster Mode

Validated the mode distinction:

  • native: Flink's native Kubernetes mode - dynamically allocates and deallocates TaskManagers as jobs submitted/completed
  • standalone: Static number of TaskManagers defined via spec.clusterSpec.taskManager.replicas

Design Feedback

Ensure UI clearly indicates mode (native vs standalone) in compute pool list and details. Users need to understand why TaskManager count changes (native) vs stays static (standalone).

Statement Lifecycle on Shared Pools

Reviewed constraints and behaviors:

  • Update Compute Pool: Rejected if statements running on shared pool. States preventing update: PENDING, FAILING, RUNNING (but not FAILED, SUSPENDED).
  • Delete Compute Pool: Only allowed if no statements running (same states as update prevent deletion).
  • Statement Submission: Shared CP must be in RUNNING phase to accept new statements.

Impact & Outcome

Cost Optimization

Low-throughput SQL statements can share pre-existing clusters rather than spawning dedicated Flink deployments.

Lower Latency

Time-to-first-result dramatically reduced since cluster is pre-warmed. Dedicated pools require cluster startup (~1-2 minutes).

Enterprise Access Control

RBAC pattern balances granularity (users see only their statements) with admin needs (full cluster visibility).

Key Learnings

Access Control Tradeoffs

Perfect granularity (every user sees only their data) is appealing but complex to implement and maintain. Sometimes "good enough" permissions (users who can access pool see all statements on it) are better tradeoff between security and implementation cost.

State Management Complexity

Shared resources (compute pools used by multiple statements) require careful state management. Preventing updates/deletes while statements are running avoids data loss and unexpected failures, but adds user friction. Clear error messages become critical.

Design Review Value

My contribution was validating that the RBAC approach made sense for enterprise deployments and that state transitions were safe. Sometimes design review is less about proposing alternatives and more about confirming assumptions with real-world deployment scenarios.
