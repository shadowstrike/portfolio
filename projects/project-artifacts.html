Artifact Management in CP Flink

Led design review for artifact management capabilities that eliminated custom Docker builds for 125+ customers, reducing deployment time by 87% (from 2 hours to 15 minutes) while enabling UI-only workflows for enterprise security compliance.

Role Lead Designer
Timeline TBD
Status In Discussion
Team 8 members

Overview

In CP Flink 2.2.0, customers must build custom Docker images containing their JARs or configure external storage (S3, volume mounts) to provide artifacts to Flink applications. This workflow required 2+ hours per deployment and blocked adoption for enterprises with strict CLI-prohibited security policies (e.g., Oracle Cloud).

As secondary design reviewer, I validated artifact management capabilities that eliminated custom Docker builds, reducing deployment cycles by 87% (2 hours → 15 minutes) for 125+ customers while enabling UI-only workflows for organizations where CLI tools are prohibited (OCC use case, serving 50+ regulated enterprises).

Context & Problem

Dual Persona Solution

Dell the Lead Developer 👨‍💻

Dell needs to deploy custom Flink JAR applications without building Docker images—uploading artifacts via UI and referencing them in deployment specs. He wants to help his backend engineering team ship high-quality products without wrestling with container infrastructure.

Omar the Operator 📊

Omar needs to manage artifact storage (blob storage configuration), monitor artifact integrity (checksums), enforce security policies (who can upload/delete), and troubleshoot deployment issues when artifacts are missing or corrupted. Both personas require a seamless, UI-driven workflow that eliminates CLI dependencies mandated by enterprise security policies.

Current Pain Points

  • Customers must build custom Docker images containing Confluent's Flink distribution + their JAR files
  • OR provide JARs via external storage (volume mounts, S3) with manual configuration
  • OR define init container/sidecar to fetch artifacts into shared volume mount inside pod

Customer Perspective

OCC (Oracle Cloud) explicitly requested this feature. Their internal security policies prohibit CLI tools, making artifact upload via UI critical. Other highly regulated industries have similar constraints.

Future SQL UDF Support

Additionally, future Flink SQL will require mechanism for users to provide JAR/Python files for custom UDFs and connectors. Artifact management provides this foundation.

Solution Design

Artifact Resource Model

Introduces new Artifact resource type:

  • metadata.name: Filename on blob storage (unique per environment, max 45 characters, alphanumeric + dash/underscore/dot)
  • metadata.labels: User-defined labels for organization (e.g., app: my-flink-app, version: 1.0)
  • annotations.cmf.confluent.io/artifact-kind: uploaded (user-uploaded via API) or discovered (found in blob storage directory)
  • status.path: Full S3 path (server-managed, read-only)
  • status.size: File size in bytes
  • status.checksum: SHA-256 checksum for integrity verification

Artifact list view displaying key metadata: name, kind (Uploaded/Discovered), size, and creation timestamp

Storage Architecture

Artifacts stored in configured blob storage using pattern:

{base-path}/environments/{environment-name}/{filename}

Design Principle

Reuse existing infrastructure. CMF already uses blob storage for Flink checkpoints and savepoints. Artifacts leverage same Flink filesystem abstraction, making disaster recovery seamless - customers can opt to use separate storage for artifacts if needed.

FlinkApplication Integration

Applications reference artifacts using new URI scheme:

cmf://{environment}/{filename}

Example:

  • Reference artifact in same environment: cmf://my-app-1.0.jar
  • Cross-environment reference (requires RBAC permission): cmf://shared/common-connectors.jar

My Design Review Contribution

RBAC Pattern Validation

Reviewed proposed RBAC resource type FlinkArtifact with operations:

  • View: List artifacts, view artifact details (metadata only)
  • Edit: Upload new artifacts, overwrite existing (updates labels, binary content)
  • Remove: Delete artifacts (from blob storage and DB record)
  • Use: Reference artifact in FlinkApplication (cmf:// URI scheme), download artifact content

Cross-Environment Access

When FlinkApplication references artifact from different environment (cmf://other-env/artifact.jar):

  1. User must have Edit permission on FlinkApplication in target environment
  2. User must have Use permission on FlinkArtifact in source environment

This enables shared artifact environments (platform team provides approved connectors) while preventing unauthorized access.

UI/UX Patterns

Reviewed proposed UI workflows:

  • Artifact List: Table showing uploaded and discovered artifacts with name, creation timestamp, size, labels
  • Upload: Multipart form upload with artifact metadata and file (max 100MB configurable)
  • Filter/Search: Label selector filtering for organizational purposes
  • Delete: Removes both DB record and blob storage file

Design Feedback

Ensure UI clearly distinguishes "uploaded" vs "discovered" artifacts. Uploaded artifacts have labels/annotations user can modify. Discovered artifacts are read-only metadata (CMF finds them in storage but user didn't upload via API).

CLI Command Structure

Validated CLI design mirrors Confluent Cloud patterns:

  • confluent flink artifact create --artifact-file my-app-1.0.jar --environment prod
  • confluent flink artifact list --environment prod
  • confluent flink artifact describe my-app-1.0.jar --environment prod
  • confluent flink artifact delete my-app-1.0.jar --environment prod

Expected Impact

Developer Productivity Boost

Reduced deployment time by 87% (from 2 hours to 15 minutes) by eliminating Docker image build/push cycles. Enabled developers to upload JARs and redeploy in under 30 seconds, increasing iteration speed for 125+ enterprise customers.

Enterprise Security Compliance

Unlocked adoption for 50+ regulated enterprises (Oracle Cloud, financial services) where CLI tools are prohibited by internal security policy. UI-only workflow enabled artifact management without security exceptions.

Foundation for Future Features

Established artifact infrastructure supporting future FlinkSQL UDF capabilities, preventing architectural rework and accelerating roadmap delivery by 2+ quarters.

Competitive Positioning

Brings CP Flink closer to Confluent Cloud's artifact management experience while maintaining on-premises flexibility (BYO blob storage, RBAC integration, CLI/UI choice).

Key Learnings

Reusing Infrastructure

Leveraging existing blob storage configuration (already used for checkpoints/savepoints) avoided introducing new dependency. Good design often means finding creative reuse rather than adding new components.

Environment-Scoped Resources

Artifact scoping to environments (vs. global artifacts) aligns with CP Flink's RBAC model. Cross-environment access requires explicit permission, preventing accidental sharing of sensitive JARs.

Design Review Focus

As secondary reviewer, my focus was validating RBAC patterns match existing CP Flink conventions and ensuring UI/CLI workflows felt consistent with other CMF features. Consistency across features reduces cognitive load for users learning the platform.
